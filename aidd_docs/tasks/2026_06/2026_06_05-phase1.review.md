## Code Quality Checklist

### Findings détaillés

- [🟡] **F-1 — Gestion d'erreur incomplète : `on Exception` ne capture pas `StateError`**
  `apps/digiharmony_app/lib/accueil/bloc/accueil_bloc.dart:68`
  `_onDemarre` enveloppe `_database.conseilDuJour(...)` dans un `try { ... } on Exception { emit(AccueilErreur()) }`. Or `AppDatabase.conseilDuJour` lève **`StateError`** (`app_database.dart:126`, « Aucun conseil seedé ») — un `Error`, **pas** un `Exception`. Si la base est ouverte mais vide (seed raté, migration partielle), le `StateError` **échappe** au `catch` → exception non gérée dans le handler du bloc au lieu du fallback État A attendu (AC7). Le test bloc ne couvre que `Exception('Drift error')` (`accueil_bloc_test.dart:147`), donc le trou n'est pas détecté.
  À noter l'**incohérence interne** : le `DemarrageBloc` traite la même source d'erreur avec `on Object` (`demarrage_bloc.dart:50`) et son test la simule justement avec `StateError` (`demarrage_bloc_test.dart:85`). Les deux blocs doivent traiter le même risque de la même façon.
  *Correctif décrit* : élargir le `catch` de `AccueilBloc._onDemarre` en `on Object` (ou `catch (e)`) pour couvrir `StateError`, et ajouter un cas de test bloc avec `thenThrow(StateError(...))` attendant `AccueilErreur`.

- [🟡] **F-2 — Code mort / commentaire trompeur : champ `_humeurSub` jamais réellement abonné**
  `apps/digiharmony_app/lib/accueil/bloc/accueil_bloc.dart:37,47-48,75`
  Le champ `StreamSubscription<EntreeHumeur?>? _humeurSub` n'est **jamais assigné** à une vraie souscription : la consommation du stream est entièrement gérée par `emit.forEach(...)`. Les lignes `await _humeurSub?.cancel(); _humeurSub = null;` (47-48) et `await _humeurSub?.cancel();` dans `close()` (75) opèrent toujours sur `null`. Le commentaire « Restartable : annule l'abonnement précédent (HB-8) » (l.41) **décrit un comportement qui n'existe pas** : aucune annulation manuelle n'a lieu, c'est `emit.forEach` qui assure le cycle de vie. Risque de fausse confiance lors d'une future évolution (restart réel).
  *Correctif décrit* : supprimer le champ `_humeurSub` et les `cancel()` associés, ou — si la restartabilité manuelle est voulue — abonner explicitement le stream à `_humeurSub` au lieu d'utiliser `emit.forEach`, et aligner le commentaire sur le code réel.

- [🟡] **F-3 — i18n : chaînes UI en dur (titres de destinations + tooltip)**
  `apps/digiharmony_app/lib/accueil/view/accueil_view.dart:86,94,107,209,210`
  `apps/digiharmony_app/lib/accueil/widgets/carte_humeur.dart:78,86,171`
  Plusieurs libellés français codés en dur sont passés à `ouvrirPlaceholder(context, '...')` puis **affichés à l'écran** par `PlaceholderScreen` (titre dans l'`AppBar` ET au centre, `placeholder_screen.dart:38,46`) : `'Choisis ta bulle'`, `'Conseil du jour'`, `'Faire une pause'`, `'Réglages'`, `'Noter mon humeur'`, `'Mon journal'`. Plus le `tooltip: 'Réglages'` (lu par les lecteurs d'écran). Ce sont des chaînes UICE visibles, non passées par l'ARB → violation de la règle « toute chaîne UI passe par l'ARB, 8 langues » (coding-assertions). En grec/roumain/turc/macédonien, l'utilisateur verra du français.
  *Correctif décrit* : ajouter des clés ARB pour chacune de ces destinations (8 langues) et résoudre le titre/tooltip via `context.l10n.*` avant de le passer à `ouvrirPlaceholder` / `IconButton.tooltip`.

- [🟢→🟡] **F-4 — Duplication `HaloRespirant` entre Demarrage et Accueil**
  `apps/digiharmony_app/lib/demarrage/widgets/halo_respirant.dart` (classe `HaloRespirant`, halo 240px, `animer` injecté, dégradé radial cyan)
  `apps/digiharmony_app/lib/accueil/widgets/halo_respirant.dart` (classe `HaloRespirant`, halo 340px, lit `MediaQuery.disableAnimations` lui-même, dégradé cyan→signature→fond)
  Deux widgets **portant le même nom de classe** mais à API et rendu divergents (taille, gradient, source du flag d'animation). Ce n'est pas une duplication 1:1 (les deux halos sont visuellement distincts), donc un partage n'est pas trivial — mais le **nom identique** est un piège de maintenance/import et brouille la lecture. Sévérité réelle : low/medium selon l'intention design.
  *Correctif décrit* : soit factoriser un widget paramétré commun (taille, gradient, `animer`) dans `lib/theme/` ou `lib/common/widgets/`, soit, au minimum, renommer pour lever l'ambiguïté (ex. `HaloMarque` pour le splash, `HaloAccueil` pour l'accueil). Décider explicitement si les deux halos doivent partager le même rendu.

- [🟢] **F-5 — Incohérence du `disableAnimations` capté une seule fois au mount (Demarrage)**
  `apps/digiharmony_app/lib/demarrage/view/demarrage_view.dart:37-47`
  `didChangeDependencies` calcule `dureeMinimale` à partir de `disableAnimations` une seule fois (garde `_demarre`). Si l'utilisateur change le réglage système pendant le splash (cas extrême, < 2,5 s), la durée n'est pas recalculée. Acceptable vu la fenêtre ; signalé pour traçabilité. *Correctif* : aucun requis ; à documenter si besoin.

- [🟢] **F-6 — Commentaire de doc obsolète dans `theme.dart`**
